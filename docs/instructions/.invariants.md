# Meeovi System Invariants

These are the rules that must always remain true across the entire Meeovi platform.  
Breaking an invariant is considered a critical error.

---

## 1. Security Invariants

- Tokens are never stored in localStorage or sessionStorage.
- All tokens are HTTP-only cookies.
- Refresh tokens are rotated on every use.
- Backend enforces all permissions.
- Frontend is always untrusted.
- No raw provider errors reach the frontend.

---

## 2. Architecture Invariants

- All external systems are accessed through adapters.
- Adapters contain no business logic.
- Capability layer contains all business logic.
- Frontend never calls providers directly.
- Gateway API validates all input.
- Gateway API normalizes all output.

---

## 3. Data Invariants

- All input validated with Zod.
- All output normalized to DTOs.
- No null or undefined values in API responses.
- No leaking provider-specific fields.

---

## 4. Search Invariants

- All search queries are sanitized.
- Pagination is always bounded.
- No wildcard or unbounded queries.
- ACLs enforced at capability layer.

---

## 5. Error Invariants

- All errors follow MeeoviError format.
- No raw provider errors.
- No stack traces exposed to frontend.
- All adapter errors are wrapped.

---

## 6. Observability Invariants

- Every capability method logs:
  - correlation ID
  - latency
  - provider used
  - success/failure
- Metrics emitted for:
  - latency
  - throughput
  - error rate
  - provider health

---

## 7. Performance Invariants

- All adapter calls have timeouts.
- All external calls use circuit breakers.
- No synchronous blocking operations.
- No unbounded loops or recursion.

---

These invariants define the non-negotiable rules of the system.  
All code must uphold them.
