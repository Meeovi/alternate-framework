# Meeovi Architecture Specification

Meeovi is a backend‑agnostic, adapter‑driven social commerce platform.  
This document defines the architectural rules that all contributors — human or AI — must follow.

---

## 1. Core Principles

- **Backend‑agnostic**: No provider‑specific logic in the frontend or capability layer.
- **Adapter‑driven**: All external systems (auth, search, commerce, social graph) are accessed through adapters.
- **Capability‑layer controlled**: Business logic lives in the capability layer, never in adapters.
- **Unified API**: Frontend interacts only with the Meeovi Gateway API (REST + GraphQL).
- **Security-first**: Backend owns authentication, authorization, and validation.
- **Predictable patterns**: All services follow the same structure, naming, and error-handling rules.

---

## 2. Layered Architecture

### Frontend (Nuxt)
- Uses composables for all API access.
- Never calls external providers directly.
- Never stores tokens in JS-accessible storage.
- Contains no business logic.

### Gateway API
- Exposes REST endpoints for auth and user.
- Exposes GraphQL for search, commerce, and social.
- Validates all input.
- Normalizes all output.
- Enforces permissions.

### Capability Layer
- Implements business logic.
- Validates inputs and outputs.
- Applies rate limits, timeouts, and error normalization.
- Calls adapters through strict interfaces.

### Adapters
- Contain provider-specific logic only.
- Must implement defined interfaces exactly.
- Must not contain business rules.
- Must sanitize provider responses.

---

## 3. Contracts and Interfaces

- All capability interfaces live in `/contracts`.
- Adapters must implement these interfaces without modification.
- Public interfaces must not be changed without explicit architectural approval.

---

## 4. Data Validation

- All incoming data is validated using Zod schemas.
- All outgoing data is normalized to DTOs.
- No raw provider responses are returned to the frontend.

---

## 5. Security Rules

- Access tokens: short-lived, HTTP-only cookies.
- Refresh tokens: rotating, device-bound.
- No tokens in localStorage or sessionStorage.
- Backend enforces all permissions.
- Search queries must be sanitized and bounded.

---

## 6. Error Handling

- All errors must be normalized to MeeoviError format.
- Provider errors must never leak to the frontend.
- Adapters must catch and wrap all exceptions.

---

## 7. Observability

- All capability methods must log:
  - correlation ID
  - latency
  - provider used
  - success/failure
- Metrics must be emitted for:
  - error rate
  - throughput
  - latency
  - provider health

---

## 8. Forbidden Patterns

- No business logic in adapters.
- No provider calls in frontend.
- No direct database access from frontend or adapters.
- No bypassing capability layer.
- No unvalidated input.
- No unbounded search queries.
