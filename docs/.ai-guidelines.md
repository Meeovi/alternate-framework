# AI Contribution Guidelines for Meeovi

This document defines how AI systems must behave when generating or modifying code in the Meeovi repository.

---

## 1. Obey the Architecture

AI must follow all rules in `.meeovi-architecture.md`.  
If a request conflicts with architecture rules, AI must warn the user and request clarification.

---

## 2. Do Not Modify Public Contracts

- Do not change capability interfaces.
- Do not change adapter interfaces.
- Do not change DTOs or schemas.
- Do not rename or move core architectural files.

Unless explicitly instructed by the human architect.

---

## 3. Follow Existing Patterns

When generating code, AI must:
- Match existing naming conventions.
- Match folder structure.
- Match error-handling patterns.
- Match logging and metrics patterns.
- Match adapter structure.

AI must not invent new patterns unless explicitly asked.

---

## 4. Tests Are the Source of Truth

- AI must not modify tests unless explicitly instructed.
- AI must implement code that satisfies existing tests.
- New features must include new tests.

---

## 5. Security Requirements

AI must ensure:
- No tokens stored in JS-accessible storage.
- All input validated with Zod.
- All provider responses sanitized.
- All adapter calls wrapped in try/catch.
- All external calls have timeouts.

---

## 6. Code Quality Requirements

AI must:
- Use strict TypeScript.
- Avoid `any`.
- Avoid unused variables.
- Avoid dead code.
- Follow ESLint rules.
- Ensure code compiles and builds.

---

## 7. Forbidden Actions

AI must not:
- Introduce new dependencies without approval.
- Modify architecture files.
- Change public APIs.
- Add business logic to adapters.
- Add provider logic to frontend.
- Generate untested code.

---

## 8. When Unsure

If AI is uncertain:
- Ask for clarification.
- Propose options.
- Never guess.

---

AI must act as a predictable, rule-bound contributor.  
Humans remain the architects.
